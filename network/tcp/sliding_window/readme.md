TCP 滑动窗口主要解决三大问题，区分流量控制、传输效率、拥塞控制。

## 1.解决「发送方发送过快，接收方缓冲区溢出」→ 流量控制
接收端缓冲区大小有限，如果发送端一味高速发送，接收端来不及上层应用读取数据，缓冲区会被占满，最终导致丢包、数据损坏。
- 原理：接收方在 ACK 报文中携带接收窗口 (rwnd)，告诉发送方自己剩余缓冲区容量。
- 规则：发送方最多只能发送 rwnd 大小的数据，窗口随接收方读取数据向后滑动、动态更新。
- 效果：强制匹配收发双方速率，防止接收端被打爆。

## 2. 解决「停等协议带宽利用率极低」→ 提升传输效率
最简单的可靠传输是停等协议：发一个报文 → 等待 ACK → 再发下一个。
- 在长链路、高 RTT 场景下，链路大部分时间处于空闲，带宽严重浪费。
- 原理：滑动窗口允许连续发送一批未收到 ACK 的报文（流水线发送），无需逐个等待应答。
- 效果：充分占用通信链路，大幅提升带宽利用率。

## 3. 配合拥塞窗口，解决「网络链路拥堵」→ 拥塞控制
流量控制只管接收端，拥塞控制管整条网络链路（路由器 / 交换机）。
网络中间设备队列溢出会引发大范围丢包、延迟飙升。
- 原理：发送方维护拥塞窗口 (cwnd)，根据丢包、RTT 变化判断网络状态，动态缩放窗口。
- 发送上限：可发送数据量 = min(rwnd, cwnd)
- 效果：根据网络拥堵程度自动调速，避免全网拥塞。
