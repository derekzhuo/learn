# 二、两种内存访问路径文字说明
## 1. 同NUMA节点·本地内存访问（推荐业务部署方式）
以DPIC Daemon运行在 `CPU2（Node0）` 举例：
1. Daemon发起共享内存/DMA缓冲区读写请求
2. 请求直接经由Socket0的CPU内置内存控制器，访问Node0本地内存
3. 数据无需经过UPI总线，无跨CPU缓存同步开销
4. 单次访问稳定延迟：40~70ns，时序无抖动、吞吐最高

适用场景：
Daemon、Coemu IP、TestBench高频交互进程全部绑定Node0偶数CPU，同时通过`numactl --membind=0`强制内存仅从Node0分配。

## 2. 跨NUMA节点·远程内存访问（业务避坑场景）
Daemon运行在 `CPU2（Node0）`，但业务内存分配在了Node1：
1. CPU2发起内存读写，页表查询发现内存归属Socket1的Node1
2. 内存请求通过UPI高速总线转发给第二颗物理CPU
3. Socket1通过自身内存控制器读取Node1本地内存数据
4. 数据原路经UPI总线返回给Socket0的CPU2
5. 额外开销：UPI总线带宽占用、跨CPU全局缓存一致性同步
6. 单次访问延迟：90~160ns，高并发下可达本地访问的3倍，极易产生随机时序毛刺

# 三、硬件关键信息汇总
1. 物理CPU数量：主板搭载**2颗独立Xeon Gold 6240R处理器**，对应2个Socket
2. 单CPU物理核心：24核，总物理核心：24 × 2 = 48核
3. 超线程状态：关闭，1个物理核心 = 1个逻辑CPU编号
4. NUMA节点绑定规则：
   - NUMA Node0（Socket0）：所有偶数逻辑CPU（0/2/4…46）
   - NUMA Node1（Socket1）：所有奇数逻辑CPU（1/3/5…47）
5. 互联通道：两颗CPU依靠UPI高速总线完成跨节点数据交互

# 四、业务最优部署拓扑文本规划
## NUMA Node 0（偶数CPU：2、4、6）｜业务专属节点
- CPU 2：DPIC Daemon 主进程（报文收发、时序录制回放）
- CPU 4：Coemu IP 仿真进程
- CPU 6：TestBench 激励下发进程
- 内存约束：强制仅从Node0分配内存，杜绝跨NUMA远程访问

## NUMA Node 1（全部奇数CPU）｜后台隔离节点
- 日志采集、监控、定时脚本
- 代码编译、打包、文件传输任务
- 终端命令、临时调试进程
作用：和仿真业务资源物理隔离，避免CPU、内存、UPI总线争抢导致时序抖动。
