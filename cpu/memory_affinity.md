# 一、什么是内存亲和（Memory Affinity）
CPU亲和性：**约束进程只能跑在哪些CPU核心上**；
内存亲和性：**约束进程只能从哪些NUMA节点分配内存**。

Linux默认内存策略：进程在哪颗CPU（哪个NUMA节点）运行，就优先在当前节点分配内存；一旦进程被OS调度跨NUMA节点，就会出现「CPU在A节点、内存却在B节点」的糟糕场景，触发高延迟的跨NUMA远程访问。
内存亲和就是通过配置，**强制把进程的内存固定分配在指定NUMA节点，无论进程被调度到哪颗CPU，都不会去其他节点申请内存**，从根源杜绝跨NUMA内存访问。

## 结合你双路48核服务器举例子
你的机器：
- NUMA Node0：偶数CPU + 第一颗CPU挂载的内存
- NUMA Node1：奇数CPU + 第二颗CPU挂载的内存

1. 只绑CPU（CPU亲和），不做内存亲和：
你把DPIC Daemon绑定在Node0的CPU2运行，但是如果Node0内存不足，操作系统依然会把内存分配到Node1；哪怕进程永远不跨CPU调度，每次读写内存都要走UPI跨NUMA访问，延迟直接翻倍。

2. CPU亲和 + 内存亲和同时配置：
```bash
numactl --cpunodebind=0 --membind=0 taskset -c 2 ./dpic_daemon
```
- `--cpunodebind=0`：进程只能在Node0所有偶数CPU上运行
- `--membind=0`：**内存亲和配置**，进程所有堆、共享内存、DMA缓冲区，只能从Node0节点的内存里申请；Node0内存不够时，进程直接OOM崩溃，绝对不会分配到Node1。

# 二、Linux 四种常用内存亲和分配策略
## 1. `--membind` 绑定模式（你的EDA仿真最优方案，强内存亲和）
- 含义：强制内存只能从你指定的NUMA节点分配，不允许分配到其他节点
- 特点：最严格，彻底杜绝跨NUMA远程访问；本地内存耗尽直接申请内存失败，不会偷偷跨节点分配
- 适用：DPIC Daemon、Coemu、TestBench这类低延迟、时序敏感的高频共享内存业务
```bash
# 内存只能从node0分配
numactl --membind=0 ./dpic_daemon
```

## 2. `--localalloc` 本地优先策略（系统默认策略）
- 含义：优先在当前进程运行的NUMA节点分配内存，本地不够就去其他节点分配
- 风险：一旦本机内存吃紧，内存会自动跑到远端节点，大概率触发跨NUMA高延迟，不适合你的仿真业务

## 3. `--preferred` 首选节点策略
- 含义：优先从指定节点分配内存，该节点内存不足时，允许在其他节点分配
- 折中方案：性能尽可能最优，同时避免OOM，延迟稳定性弱于membind

## 4. `--interleave` 交错分配策略
- 含义：内存轮询分散分配在多个NUMA节点
- 场景：大带宽批量数据运算，需要均衡多节点内存带宽；**绝对不能用于你的低延迟仿真业务**，会持续跨NUMA访问，延迟抖动严重。

# 三、为什么必须同时配置CPU亲和 + 内存亲和？两个典型踩坑场景
### 场景1：CPU固定在Node0，内存自动分配到Node1
只绑CPU不限制内存，Node0内存不足时，操作系统会把共享内存、DMA缓冲区分配到Node1。
进程一直在Node0的CPU运行，但每次读写内存都要走UPI总线访问远端内存，延迟翻倍，MCLK时序、报文处理出现大量毛刺。

### 场景2：内存一开始在Node0，进程被调度到Node1的CPU
默认本地分配策略下，进程启动在Node0，内存分配在Node0；后续操作系统把进程调度到Node1奇数CPU，此时CPU在Node1，内存还在Node0，全程远程内存访问。
哪怕你绑了CPU核心，如果绑的CPU跨NUMA，依然会踩坑。

# 四、适配你业务的最佳实践
1. 同一组高频交互进程（Daemon、Coemu、TestBench）统一：
   - CPU绑定：全部NUMA Node0（偶数CPU）
   - 内存亲和：`--membind=0`，强制内存只从Node0分配
2. 后台日志、编译、监控等低优先级任务统一放到Node1，CPU+内存都绑定node1，资源完全隔离；
3. 启动命令模板：
```bash
# CPU限制在node0，内存强制分配在node0，绑定CPU2、4、6
numactl --cpunodebind=0 --membind=0 taskset -c 2,4,6 ./runtime_sw_service
```

# 五、CPU亲和 vs 内存亲和 一句话区分
- **CPU亲和**：约束「程序在哪上班」（固定工位）
- **内存亲和**：约束「程序去哪拿数据资料」（固定仓库）
最优方案：让程序**在同一个办公楼上班、同一个仓库拿资料**（CPU+内存同NUMA节点），既不换工位，也不跨楼取资料，延迟最低、时序最稳定。
